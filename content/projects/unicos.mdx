## Product context and current status

Unicos is a body-shop workflow application in active development. It organizes customer and vehicle intake, repair orders, estimates and supplements, billing, payments, attachments, communication entries, and operational reports around the repair order as the central record. That center matters because estimates, invoices, payments, attachments, and communication history derive their workflow context and access boundaries from the repair order, while repair-order status governs final-state and billing restrictions.

The current application runs only as a local-development Django and PostgreSQL system through Docker Compose. It is not production-deployed or production-ready, and it is not presented as evidence of real-shop operation. The implemented work demonstrates domain modeling, authorization decisions, workflow services, and validation practices inside an actively developed product. Production hosting, security hardening, media infrastructure, backups, monitoring, and operational readiness remain separate future work.

I owned the requirements, workflow constraints, architecture decisions, slice approvals, review, validation, and final merge decisions. AI tools accelerated source inspection, implementation, testing assistance, and diff review. Human review retained control over requirements, scope, domain decisions, evidence interpretation, and acceptance; the work is not represented as independently written line by line without AI assistance.

## Server-rendered architecture

Unicos is structured as a server-rendered Django monolith backed by PostgreSQL. Django views, forms, templates, models, and service modules live in one application boundary, which keeps a workflow change close to the authorization checks and business invariants it affects. Docker Compose provides the current development environment for the web process and database. That configuration uses Django's development server and is not a production deployment design.

The interface uses locally stored Bootstrap assets with project-specific CSS overrides. HTMX is limited to places where progressive enhancement improves a focused interaction, currently including repair-order status updates. The normal form POST and redirect path remains available, so the status workflow does not depend on client-side application state. This kept the product within Django's request-response model without adding a frontend framework or frontend build pipeline before the current workflow required one.

## Repair-order domain spine

The implemented domain direction can be summarized as `Customer -> Vehicle -> Repair Order -> Estimate or Supplement -> Draft Invoice -> Invoice and Payment`. A repair order links the customer, vehicle, assigned staff member, workflow status, status history, estimates, communications, attachments, and billing record. Repair-order, estimate, and invoice numbers are generated by the application using generic yearly sequences rather than being entered manually.

Estimates and supplements are related but distinct records. A supplement is modeled as an estimate with a parent from the same repair order, giving additional work its own line items and lifecycle without implying a complete estimate-versioning system. An eligible approved, itemized estimate can be converted to a draft invoice. The conversion links the invoice to its source estimate and copies the line items, but it does not issue the invoice or record a payment.

Each repair order currently has at most one invoice. Communication entries capture repair-order-scoped history without sending messages, while attachments use policy-gated local file storage. These choices support the current workflow but do not provide complete insurance-claim management, parts operations, scheduling, generated documents, or supplement-to-invoice merging.

## Authorization and object policy

Authentication uses a custom staff user model. Named shop roles feed a centralized application permission matrix, and shop-facing views enforce those permissions on the server. Templates may hide unavailable actions for clarity, but a hidden button is not the authorization boundary. The underlying view and policy checks remain authoritative.

Object policies add a second boundary around repair orders and the estimate, invoice, payment, attachment, and communication workflows that inherit repair-order scope. A technician-only user can view assigned repair orders but not unrelated work. Estimate and billing access follows the associated repair order rather than becoming a separate unrestricted path. Django's `is_staff` flag is deliberately not treated as a shop-facing permission.

This is meaningful object-level authorization, but it is incomplete. Full customer and vehicle object policies are not implemented, admin authorization is not unified with shop-facing policy helpers, and there is no multi-shop tenancy. Django admin remains a separate trusted emergency and development interface. For those reasons, the current design should not be described as complete RBAC or complete object-level authorization.

## Lifecycle and business-rule services

Repair-order status transitions pass through a centralized lifecycle service. The service checks business rules before applying a transition and records the resulting status history through the model boundary. A repair order cannot move into a final state while a non-void invoice still has a balance due. Because this check belongs to the service rather than a template, shop-facing views and aligned admin operations receive the same rule.

Estimate transitions are similarly explicit: a draft may be submitted, and a submitted estimate may be approved, declined, or expired. Submitted and terminal estimates become read-only in normal workflows. Approval is not just a status value; the transition requires an approval source and contact before the server records the approval timestamp.

Conversion is also guarded. The estimate must be approved, have captured approval evidence, contain line items, have no prior conversion, and belong to a repair order without an invoice. The service locks the estimate, creates a draft invoice, and copies line items inside one database transaction. It intentionally leaves invoice issue as a separate decision. These boundaries keep workflow invariants in services and models rather than relying on page presentation.

## Billing and transaction boundaries

Billing services centralize invoice issue and void actions, payment recording, and invoice line-item creation and editing. They reject billing mutations for final repair orders and prevent edits or payments on void invoices. Payment validation rejects non-positive values and overpayment, while invoice payment status is synchronized with recorded payments.

Concurrent payment attempts are a specific risk because two requests could both inspect the same remaining balance. Payment recording therefore runs in a transaction and locks the invoice row with `select_for_update()` before rechecking eligibility and saving the payment. Estimate conversion uses a transaction and row lock for the related duplicate-conversion risk as well.

These controls protect the current billing workflow, but Unicos is not a full accounting ledger or payment-processing platform. It records application-level invoices and payments; it does not integrate with a payment processor, accounting system, or production financial operations environment.

## Admin service alignment

Django admin is retained for trusted inspection and emergency or development correction, not as the normal shop-facing workflow. Earlier direct admin mutations could bypass services that protected estimate approval, invoice lifecycle, payments, line items, and attachment evidence. The aligned admin pathways now call the same domain services for major mutations or make risky fields and inline operations read-only.

That reduces accidental invariant bypass without claiming that admin is fully unified with the application policy model. There is no audited override path, no complete admin object-policy integration, and some approval metadata text remains directly editable. Admin therefore stays a separate trusted interface with explicit limitations rather than a second equivalent operator application.

## Validation and delivery

Development proceeds through narrow feature slices with structured review and human-controlled acceptance. CI separates Ruff lint and formatting checks from Django and PostgreSQL-backed validation. It runs Django system checks, verifies migration consistency, applies migrations, executes the test suite against PostgreSQL, builds the Docker image, starts the Compose environment, and checks the public `/healthz/` endpoint for local and CI runtime validation.

Manual browser review supplements those checks for shop-facing workflows where page state, available actions, and status transitions matter. The health endpoint confirms that the local application process responds; it is not production monitoring. Automated checks cover important behavior without establishing universal test coverage, and merge decisions remain human-controlled even when AI assists with implementation and analysis.

## Tradeoffs and current boundaries

The server-rendered monolith favors direct domain workflows and a small operational surface. Limited HTMX adds targeted responsiveness without creating a second frontend state model. That simplicity is useful during active development, though future interaction needs may justify different boundaries.

One invoice per repair order keeps current billing rules understandable but limits supplement conversion and multi-estimate merging. Local `FileField` media persistence supports development evidence, not production storage, private serving, scanning, or recovery. Browser printing avoids a server-generated PDF subsystem, but it is not generated-document infrastructure. Read-only filtered reports and CSV exports provide operational snapshots rather than revenue analytics, receivables aging, or mature business intelligence.

Production hardening is its own body of work. A hosted path would require production web serving, hardened settings, HTTPS, deployment-specific secrets, media design, backups and restore validation, monitoring, and a stricter admin boundary. Keeping those gaps visible is part of the architecture: implemented workflow depth should not be mistaken for production readiness.
